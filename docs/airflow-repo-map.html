02

운영하는 분 — 배포에서 확인까지

코드가 merge된 뒤 DAG가 반영되고 task가 실행되는 경로입니다. 배포 설정과 application image 승격은 서로 다른 변경 표면입니다.

01GitHub main merge

DAG, helper, values 또는 image digest 변경을 main에 반영합니다.

02git-sync 동기화

dags/ subPath와 같은 revision의 DAG/helper를 scheduler에 전달합니다.

03DAG import/parse

Scheduler가 DAG를 읽고 import error와 task topology를 만듭니다.

04task 실행

KPO 또는 scheduler-owned operator가 정해진 실행 위치에서 동작합니다.

05결과·로그 생성

GCS·BigQuery·Feast 결과와 Airflow task log를 확인합니다.

06운영 검증

DAG import error, task state, pool, image digest를 대조합니다.

변경 종류를 먼저 구분합니다. DAG/helper 변경은 git-sync 반영 대상이므로 Airflow image 재빌드가 기본 조건이 아닙니다. batch 애플리케이션 변경은 Autoresearch image release와 immutable digest 승격을 거치고, Helm values 변경은 deploy/airflow chart 배포로 반영합니다.
인프라 경계: Autoresearch-infra는 GKE, IAM, 데이터셋·네트워크를 소유합니다. Airflow DAG에서 증상을 확인하더라도 node pool·Workload Identity·private network 변경은 infra 레포의 책임입니다.

04

실행 환경과 배포 구성

Airflow가 task를 관리하는 표면과, GKE·Helm이 그 실행 환경을 제공하는 표면을 나누어 봅니다.

airflow runtime
  • Scheduler — DAG parsing, schedule, dependency와 scheduler-owned operator를 관리합니다.
  • Webserver — Airflow UI와 manual trigger의 진입점입니다.
  • git-sync — main의 dags/ subPath를 scheduler에 동기화합니다.
  • KPO — batch image를 일회성 Kubernetes Pod로 실행합니다.
  • metadata DB — DAG run, task state, pool과 Airflow 메타데이터를 저장합니다.
gke / helm deployment
  • deploy/airflow/Chart.yaml — Apache Airflow 공식 chart를 감싸는 umbrella chart입니다.
  • deploy/airflow/values.yaml — 현재 dev 운영 기준입니다.
  • deploy/airflow/values.example.yaml — 신규 환경 구성용 예제입니다.
  • docker/airflow/Dockerfile — Airflow 런타임 이미지 build context입니다.
  • Workload Identity — Scheduler와 KPO가 Google 리소스에 접근하는 인증 경계입니다.
  • immutable GAR digest — AUTORESEARCH_BATCH_IMAGE가 가리키는 실행 artifact입니다.
Airflow image, DAG git-sync, Helm chart, application batch image는 모두 다른 변경 표면입니다. 변경 내용을 먼저 분류하면 불필요한 image rebuild와 잘못된 rollback 범위를 줄일 수 있습니다.

운영 명령과 용어

명령을 외우기보다 현재 서비스를 뒤에서 움직이는 공장의 어느 부분을 보는지부터 생각합니다. 아래 순서는 서비스의 문 앞 → 설계도 전달 → 처리량 게이트 → 운영자 입구를 확인하는 흐름입니다.

서비스와 연결해서 보면

Autoresearch를 유튜브 추천 공장이라고 생각하면, Airflow는 고객 요청을 직접 처리하는 추천 API가 아니라 매일 추천에 필요한 재료를 수집·가공·검수·진열하는 뒤편 관제실입니다. GCS·BigQuery·Feast는 재료 창고와 장부이고, KPO는 작업마다 호출되는 임시 작업 차량입니다.

YouTube·action-log

공장에 들어오는 원재료와 고객 행동 기록입니다.

DAG·Scheduler

오늘 어떤 재료를 어떤 순서로 처리할지 정하는 생산계획과 관제실입니다.

GCS·BigQuery

파일을 보관하는 창고와 검수·조회용 장부입니다.

Feast·Redis

검수된 feature를 빠르게 꺼내 쓰도록 진열하는 online 선반입니다.

운영 명령 — 어디가 막혔는지 찾는 순서

01서비스의 문 앞 — 클러스터와 입구가 살아 있는가

먼저 내가 어느 GKE 건물을 보고 있는지와 airflow namespace에 들어갈 권한이 있는지 확인합니다. Pod는 공장 안의 작업실이고 Service는 외부에서 찾는 접수 창구입니다.

kubectl config current-context
kubectl auth can-i get pods -n airflow
kubectl get pods -n airflow
kubectl get svc airflow-webserver -n airflow

정상 기대: 예상 context, 권한 허용, scheduler/webserver/git-sync가 Ready인 Pod, webserver Service가 보입니다.

다음 확인: Pod가 없으면 배포·node 상태를, Service가 없으면 Helm release와 namespace를 먼저 봅니다.

02설계도 전달 — DAG가 최신 코드로 들어왔는가

git-sync는 공장 작업자에게 최신 설계도를 전달하는 배달원입니다. Scheduler가 DAG를 못 찾거나 import error를 내면 task 실행보다 먼저 전달·파싱을 확인합니다.

kubectl logs -n airflow airflow-scheduler-0 -c git-sync --tail=80
kubectl exec -n airflow airflow-scheduler-0 -c scheduler -- airflow dags list
kubectl exec -n airflow airflow-scheduler-0 -c scheduler -- airflow dags list-import-errors

정상 기대: git-sync log에 대상 revision이 보이고, DAG 목록에 이름이 있으며, import error가 0건입니다.

다음 확인: revision이 오래됐으면 repo·branch·subPath를, import error가 있으면 해당 파일과 provider dependency를 봅니다.

03처리량 게이트 — 외부 API 작업이 줄을 서는가

action-log shard는 여러 작업 차량이 OpenRouter라는 하나의 톨게이트를 통과하는 구조입니다. Airflow Pool은 입구의 통과 슬롯이고, Pod 내부 concurrency는 차량 한 대가 동시에 처리하는 차선 수입니다.

kubectl exec -n airflow airflow-scheduler-0 -c scheduler -- airflow pools get action_log_openrouter
kubectl get pods -n airflow

정상 기대: action_log_openrouter가 2 slots이고, shard task가 pool 때문에 무한 대기하지 않습니다.

다음 확인: slot이 모두 사용 중이면 task를 무작정 늘리지 말고 shard 수·Pod concurrency·retry를 함께 확인합니다. 현재 최대 OpenRouter 요청 동시성은 2 × 3 = 6입니다.

04운영자 입구 — Webserver와 배포 결과를 확인하는가

Webserver는 추천 서비스 자체가 아니라 Airflow 관제실의 프런트 데스크입니다. UI 접근이 필요할 때만 localhost로 문을 연결하고, 배포 변경은 chart가 렌더링 가능한지 먼저 검사합니다.

kubectl get deploy airflow-webserver -n airflow
kubectl logs -n airflow deploy/airflow-webserver -c webserver --tail=80
kubectl port-forward -n airflow svc/airflow-webserver 8080:8080
helm lint deploy/airflow

정상 기대: Webserver Deployment가 Ready이고 log에 반복 crash가 없으며, port-forward 후 http://localhost:8080/login/에 접근할 수 있습니다.

다음 확인: port-forward는 현재 터미널을 점유하므로 확인 후 Ctrl-C로 종료합니다. Helm 오류는 cluster에 적용하기 전에 chart·values에서 수정합니다.

장애 위치를 서비스 흐름으로 좁히기

Pod가 없다

공장 건물 자체가 열리지 않은 상태입니다. namespace, Helm release, node pool과 deployment를 먼저 확인합니다.

DAG가 목록에 없다

설계도가 작업실까지 오지 않은 상태입니다. git-sync revision·branch·subPath와 Scheduler parse를 확인합니다.

DAG는 있는데 멈췄다

생산계획은 있지만 순서표가 다음 작업을 호출하지 못한 상태입니다. schedule, Dataset, dependency, Pool을 확인합니다.

task가 실패했다

작업 차량이 현장에서 멈춘 상태입니다. Airflow task log에서 batch stdout과 Pod 원인을 확인하고 image digest·Secret·Workload Identity를 구분합니다.

용어 — 정의보다 서비스에서의 역할로 이해하기
DAG

정의: task, 의존성, trigger, retry 정책을 선언한 workflow입니다. 서비스 비유: 오늘의 유튜브 재료를 수집하고 action-log를 만들고 검수하는 생산라인 도면입니다. 운영 의미: 계산 코드를 고치는지, 순서·실행 조건을 고치는지 구분하는 첫 경계입니다.

Dataset

정의: upstream 데이터 갱신을 downstream DAG의 실행 조건으로 전달합니다. 서비스 비유: 창고에 재료가 입고됐다는 호출벨입니다. 운영 의미: feast_offline_feature_build는 raw Dataset 두 개가 모두 갱신돼야 시작합니다.

KPO

정의: KubernetesPodOperator로 task마다 batch image Pod를 만듭니다. 서비스 비유: 주문 한 건마다 필요한 도구를 실은 임시 작업 차량입니다. 운영 의미: Pod가 실패하면 DAG 코드뿐 아니라 image digest, Pod ServiceAccount, Secret, node scheduling도 함께 봐야 합니다.

git-sync

정의: Airflow image와 분리된 DAG source를 scheduler Pod에 동기화하는 sidecar입니다. 서비스 비유: 중앙 설계도 보관소에서 작업실로 최신 도면을 배달하는 사람입니다. 운영 의미: DAG 변경인데 image를 다시 빌드하기 전에 revision과 dags/ subPath를 확인합니다.

Pool

정의: 제한된 자원의 동시 실행 수를 슬롯으로 제어합니다. 서비스 비유: OpenRouter 톨게이트의 통과 차단기입니다. 운영 의미: 현재 action_log_openrouter는 2 slots이고 shard 내부 concurrency 3과 곱해 최대 6개 요청을 만듭니다.

KSA / GSA

정의: Kubernetes Service Account와 Google Service Account를 Workload Identity로 연결합니다. 서비스 비유: Pod의 사원증과 GCP 시설 출입 여권입니다. 운영 의미: 키 파일을 넣지 않고도 GCS·BigQuery·Redis·Secret Manager 접근이 되는지 확인하는 인증 경계입니다.

immutable digest

정의: image를 sha256 digest로 고정한 실행 artifact입니다. 서비스 비유: 내용물을 봉인하고 봉인번호를 기록한 배송 상자입니다. 운영 의미: application image rollback과 DAG revision rollback을 서로 독립된 자산으로 선택할 수 있습니다.

reschedule sensor

정의: 기다리는 동안 worker를 점유하지 않고 다음 poke 시점에 다시 예약하는 sensor 모드입니다. 서비스 비유: 직원이 문 앞에서 계속 서 있지 않고 대기표를 받아 돌아오는 방식입니다. 운영 의미: GCS partition 도착을 기다리면서 worker를 붙잡지 않지만 timeout과 재처리 날짜는 확인해야 합니다.

metadata DB / Cloud SQL

정의: Airflow의 DAG run, task state, pool과 운영 메타데이터를 저장합니다. 서비스 비유: 관제실이 남기는 생산일지와 작업 이력 장부입니다. 운영 의미: GCS·BigQuery 데이터와 다른 저장소이며, 현재 배포는 내장 PostgreSQL 대신 private IP Cloud SQL을 사용합니다.

Webserver Service

정의: Airflow UI를 노출하는 Kubernetes Service입니다. 서비스 비유: 공장 내부 관제실로 들어가는 방문자 접수 창구입니다. 운영 의미: UI 입구가 안 보이는 문제와 실제 batch task가 실패한 문제를 분리해서 확인해야 합니다.
